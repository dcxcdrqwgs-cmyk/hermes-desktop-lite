# 会话级模型切换设计

## 目标

在聊天窗口中，为每个对话提供独立的模型选择能力。模型切换只影响当前对话后续消息，不改全局默认模型，也不影响其他会话。

这个设计对齐 Hermes 官方 `/model` 语义：默认只影响当前会话，只有 `--global` 才会修改全局默认模型。

## 用户体验

### 主入口

在聊天主视图头部，把模型选择器放在“新对话”按钮后面。

状态规则：

- 有活动会话时，选择器显示该会话当前绑定的模型。
- 没有活动会话时，选择器显示“下一条新对话将使用的模型”。
- 切换模型后，文案明确提示“仅影响当前对话后续消息”。

### 交互行为

- 用户在空白聊天页先选模型，再发送第一条消息。
  - 创建新会话时，把所选模型写入该会话。
- 用户在已有会话中切换模型。
  - 仅更新当前会话的模型。
  - 下一次发送消息前，确保 Hermes 当前会话已切到这个模型。
- 会话列表展示一个轻量模型 badge，帮助区分不同模型的对话。

### 候选模型来源

优先从“模型配置”页已配置的模型中生成候选项：

- 云端 Provider 的默认模型
- 自定义接入当前模型
- 本地模型接入当前模型

如果当前会话模型不在候选列表中，也要保留显示，避免旧会话出现“值丢失”。

## 数据设计

### 会话数据

当前 `sessions` 表已有会话级字段，但没有 `model`。新增：

- `sessions.model TEXT NULL`

语义：

- `NULL` 或空值表示“跟随当前 Hermes 默认模型”
- 非空字符串表示“该会话后续消息优先使用这个模型”

### 前端状态

前端新增两层状态：

- `pendingConversationModel`
  - 用于空白聊天页，还未创建会话时的“下一条新对话模型”
- `activeSessionModel`
  - 来自当前会话记录

切换优先级：

1. 有活动会话时，读写 `activeSession.model`
2. 无活动会话时，读写 `pendingConversationModel`

## 后端设计

### SQLite 迁移

在会话数据库初始化逻辑中增加：

- `ALTER TABLE sessions ADD COLUMN model TEXT`

要求：

- 兼容已有数据库
- 重复执行不报错

### 命令扩展

扩展现有 Tauri 命令：

- `get_sessions`
  - 返回 `model`
- `create_session`
  - 支持传入 `model`
- 新增 `update_session_model`
  - 更新当前会话模型并刷新 `updated_at`
- 视情况新增 `get_session_model`
  - 如果前端只靠 `get_sessions` 和当前会话对象即可，不必额外新增

## 发送链路设计

### 当前现状

当前发送链路固定走：

- Responses API body 中 `model: "hermes-agent"`

这意味着“按钮选了模型”还不会真的影响 Hermes。

### 推荐实现

采用“会话发送前同步模型”的方式，而不是在桌面端发明一套独立模型路由。

具体流程：

1. 用户切换当前会话模型
2. 前端保存 `sessions.model`
3. 用户下一次发送消息前：
   - 读取当前会话的 `model`
   - 如果该模型与当前 Hermes 会话模型不一致，先执行一次会话内模型切换
4. 模型切换成功后，再按原流程发送用户消息

### 模型切换执行方式

优先方案：

- 通过 Hermes 当前会话内发送 `/model <name>`

原因：

- 完全复用 Hermes 官方会话语义
- 不需要桌面端直接接管 provider 解析逻辑
- 与 CLI / slash command 一致

约束：

- 只在“下一条正式用户消息发送前”隐式执行一次
- 不把内部 `/model` 切换写进用户可见消息列表
- 切换失败时，中断发送，并提示错误 toast

### 无活动会话时

当用户在空白页选中模型后：

1. 点击发送第一条消息
2. 先创建新会话并写入 `session.model`
3. 进入该会话
4. 在正式消息发送前执行一次内部 `/model`
5. 成功后再发送用户消息

## UI 设计

### 聊天头部

在 `App.jsx` 的聊天头部新增模型选择器：

- 样式与现有 Toolbar / Dropdown 视觉一致
- 紧跟“新对话”按钮
- 文案尽量短，例如：
  - `模型: GPT-4.1`
  - `模型: qwen2.5:14b`

### 列表展示

在 Sessions 页面会话卡片里新增模型 badge：

- 仅在会话模型非空时显示
- 放在标题同一行或时间信息下方
- 不喧宾夺主

### 空态提示

当没有任何可选模型时：

- 选择器禁用
- 提示先到“模型配置”页设置默认模型

## 错误处理

- 会话模型保存失败
  - 保持当前 UI 不提交
  - toast 提示
- 内部 `/model` 切换失败
  - 本次消息不发送
  - toast 提示切换失败
  - 会话模型值保留，但提示“当前 Hermes 会话尚未切换成功”
- 当前模型不在候选列表中
  - 仍显示当前值
  - 在下拉里附加一个“当前会话模型”临时项

## 测试策略

### 前端

- 会话模型候选项构建逻辑
- 空白页与已存在会话两种状态的展示逻辑
- 模型切换后只更新当前会话，不影响其他会话
- 没有可选模型时的禁用态

### 后端

- `sessions.model` 迁移兼容已有数据库
- `create_session` 能正确写入 `model`
- `get_sessions` 能正确返回 `model`
- `update_session_model` 只影响目标会话

### 集成

- 空白页选择模型并发第一条消息，创建新会话成功
- 已有会话切换模型后再发送消息，后续消息走新模型
- 切换失败时消息不会继续发送

## 范围边界

本次不做：

- 全局模型切换 UI
- 每条消息单独选模型
- provider 级别高级参数随会话覆盖
- 对历史消息回溯标注“当时使用的模型”
- 模型能力探测、上下文长度自动同步

## 风险与注意点

- Hermes 内部 `/model` 是否能在桌面端当前 Responses 会话上下文中稳定生效，需要先做一次最小验证。
- 如果桌面端与 Hermes 的“会话”边界不完全一致，需要补一层桌面端到 Hermes 会话的映射约束。
- 如果内部命令切换会污染消息历史，需要额外把这类系统级切换从用户可见消息流中隔离。

## 推荐实施顺序

1. 先验证 Hermes 当前桌面发送链路下，会话内 `/model` 是否能稳定切换。
2. 再补 SQLite `sessions.model` 与前后端 API。
3. 然后加聊天头部模型选择器。
4. 最后补 Sessions 列表 badge 与失败提示。
